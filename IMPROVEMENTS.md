# Как улучшить MVP

Направления развития за рамками базового MVP. Для каждого: зачем нужно нашему кейсу,
какие файлы менять, как проверить улучшение.

---

## ✅ Реализовано: улучшение интерфейса (Streamlit)

**Зачем:** на корпусе из 11 000 чанков один фиксированный порог и top-k неудобны —
для одних запросов нужно больше фрагментов, для других строже отсекать шум; а читать
длинный фрагмент проще, когда видно, какие слова совпали с запросом.

**Что сделано** (`app/main.py`):
- ползунок **top-k** (1–10) — сколько фрагментов показывать;
- ползунок **порога релевантности** (`min_score`, 0.0–0.5) — прокидывается в `ask(...)`;
- **подсветка** слов запроса в найденных фрагментах (`<mark>`), с экранированием HTML.

**Как проверить:** запустить `uv run streamlit run app/main.py`, задать «инфляция и рост
тарифов», подвигать порог — лишние фрагменты сворачиваются/отсекаются; слова запроса
подсвечены в тексте.

## ✅ Реализовано (частично): качество поиска

**Зачем:** служебные слова («как», «и», «на») раздували cosine, из-за чего
нерелевантный вопрос «Как приготовить борщ?» не получал отказ.

**Что сделано** (`app/config.py`, `scripts/build_index.py`): русские **стоп-слова** +
**биграммы** (`ngram_range=(1,2)`) + `min_df=2` в TF-IDF. После этого нерелевантные
запросы дают score = 0.00 → корректный отказ, а релевантные — 0.33–0.38.

## 📋 План: эмбеддинги вместо TF-IDF

- **Сейчас:** поиск по совпадению слов (TF-IDF + cosine).
- **Улучшение:** `sentence-transformers` (например, `intfloat/multilingual-e5-base`) для поиска по смыслу.
- **Файлы:** `build_index.py` считает эмбеддинги, `retriever.py` ищет ближайшие векторы.
- **Проверка:** запросы-перефразировки («стоимость доллара» vs «курс рубля») должны находить те же статьи.

## 📋 План: векторная база

- **Сейчас:** индекс в файлах (`vectorizer.pkl`, `matrix.npz`).
- **Улучшение:** Chroma/Qdrant/FAISS для быстрого ANN-поиска по большим коллекциям.
- **Файлы:** слой хранения и `retriever.py`; процесс обновления данных.
- **Проверка:** время поиска не растёт при увеличении корпуса в 10 раз.

## 📋 Прочие направления

- **Подключение LLM:** в `generator.py` собирать ответ через внешний OpenAI-совместимый API или локальную модель по найденному контексту, сохраняя правило «только по источникам».
- **Hybrid search:** комбинировать TF-IDF/BM25 + эмбеддинги.
- **Reranking:** cross-encoder для пересортировки top-k.
- **Eval:** скрипт с набором эталонных вопросов и метриками retrieval@k / hit-rate.
- **Автообновление данных:** пересборка корпуса из свежего Gazeta по расписанию, версионирование.
