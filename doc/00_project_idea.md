# Идея проекта

**Учебный RAG по деловым новостям** — не production-система, а пример для обучения: как устроен RAG и как им пользоваться от первого запроса до ответа с источниками.

## Проблема

RAG часто воспринимают как «промпт к ChatGPT», хотя это **цепочка шагов**: данные → чанки → индекс → поиск → ответ. Без простого рабочего примера непонятно, где на каждом этапе может сломаться качество.

Отдельная боль для корпоративных данных: новостной/документный поток большой, и нужно не «угадать ответ», а **найти первоисточник** и сослаться на него.

## Цель

Собрать учебный проект, который:

- показывает **как пользоваться RAG**, а не только что это такое;
- запускается локально, без внешних API и ключей;
- проходит весь pipeline от сырых новостей до ответа;
- возвращает ответ **с источниками** (заголовок статьи, `doc_id`, фрагмент, score);
- **явно отказывается**, если в базе нет релевантных данных.

## Для кого

Студенты курса «Корпоративные данные» и начинающие разработчики, которые учатся строить RAG с нуля.

## Зачем RAG

Обычная LLM не знает ваших документов и может выдумать ответ. RAG сначала **ищет** релевантные фрагменты в базе, потом **формирует** ответ по ним:

```
вопрос → поиск в базе → контекст + вопрос → ответ со ссылками
```

**RAG = retrieval (поиск) + generation (генерация по найденному)**

## Данные

Источник — **корпус деловых и экономических новостей Gazeta.ru** (датасет `IlyaGusev/gazeta` на HuggingFace): рубрики business, financial, realty, techzone.

Почему подходит для обучения:

- открытые данные, скачиваются публично, без регистрации;
- естественная структура (заголовок, текст, рубрика, дата) — удобно для метаданных и источников;
- деловая тематика близка курсу «Корпоративные данные»;
- легко набрать 1000+ записей и 10 000+ чанков.

## Результат MVP

Streamlit-приложение, куда можно задать вопрос, например:

> «Центробанк ставка рефинансирования»

Система находит подходящие фрагменты новостей, формирует ответ и показывает, **из какой статьи** взята информация (заголовок, `doc_id`, score, текст).

## Порядок документов разработки

| Шаг | Документ | Что решаем |
|-----|----------|------------|
| 1 | `00_project_idea.md` | **Идея** — проблема, цель, данные |
| 2 | `vision.md` | **Стек и границы** — технологии MVP |
| 3 | `conventions.md` | **Правила** — как писать код |
| 4 | `tasklist.md` | **План** — итерации 0–9 |
| 5 | `workflow.md` | **Процесс** — как вести итерации |
| + | `DATA.md` | **Данные** — источник, что индексируем |

## Критерий успеха

Студент может объяснить своими словами:

1. Зачем нужен каждый шаг pipeline.
2. Почему ответ без источников ненадёжен.
3. Как проверить, что система отвечает по данным, а не выдумывает.
4. В каком порядке принимать решения при новом RAG-проекте.
